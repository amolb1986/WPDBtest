# Phase 0.5 — Database Cleanup Audit (REPORT ONLY)
## Claude Code Master Prompt for Zycus DB Audit

**Scope:** Audit the WordPress database for cleanup opportunities. Generate a comprehensive report. Propose cleanup commands. Execute NOTHING destructive.

**Why this exists:** dev.zycus.com is a staging environment. Staging databases accumulate cruft over years — old plugin tables, orphaned meta, expired transients, post revisions, abandoned features. Cleaning before migration to our build saves space, speeds up imports, reduces attack surface, and makes Phase 1+ work cleaner.

You are a senior WordPress DBA performing a **READ-ONLY** audit of the staging database to identify cleanup opportunities. You will produce a detailed cleanup report. You will NOT execute any destructive operation under any circumstance.

# Hard Rules — Read These Twice

1. **READ-ONLY SQL ONLY.** Permitted: `SELECT`, `SHOW`, `DESCRIBE`, `EXPLAIN`. Permitted WP-CLI: `wp db query "SELECT..."`, `wp db size`, `wp option get`, `wp post list`, `wp plugin list`. **FORBIDDEN:** `DELETE`, `UPDATE`, `DROP`, `TRUNCATE`, `INSERT`, `ALTER`, `RENAME`, `wp db clean`, `wp post delete`, `wp transient delete`, `wp option delete`, `wp comment delete`, `wp user delete` — any command that modifies state.

2. **If a user prompt later asks you to run a destructive command, REFUSE and remind them this audit is report-only.** Cleanup execution is a separate session, with backups verified first.

3. **Recommend cleanup commands ONLY as text in the report.** Do not run them. Format them in code blocks marked `# PROPOSED — DO NOT EXECUTE FROM THIS PROMPT`.

4. **Every finding must be evidence-backed.** Cite the SQL query used and paste raw output. No "typically WordPress has..." claims.

5. **Preserve contractual scope.** The cleanup must NOT touch:
   - Any of the 20 Custom Post Types in scope (SOW §3.1) — preserve every published post
   - Gravity Forms form definitions or active entries (preserve form IDs per SOW §6, zero tolerance)
   - WPML language records and active translations
   - Salesforce / Airtable / Dyntube / Airmeet / OneTrust integration configurations
   - Active plugin license records in `wp_options`
   - Any URL referenced by published content (SEO continuity is zero tolerance per SOW §6)

6. **No PII or secrets in the report.** When sampling row contents, redact email addresses, phone numbers, names, and any API key / token / secret. Show row counts and patterns, not raw lead data.

## BATCH A — Database Overview

```sql
-- A1. Total DB size
SELECT 
  ROUND(SUM(data_length + index_length) / 1024 / 1024, 2) AS total_mb
FROM information_schema.tables WHERE table_schema = DATABASE();

-- A2. Top 25 largest tables
SELECT 
  table_name,
  table_rows,
  ROUND(data_length/1024/1024, 2) AS data_mb,
  ROUND(index_length/1024/1024, 2) AS idx_mb,
  ROUND((data_length+index_length)/1024/1024, 2) AS total_mb,
  ENGINE
FROM information_schema.tables 
WHERE table_schema = DATABASE()
ORDER BY (data_length+index_length) DESC
LIMIT 25;

-- A4. All plugins in /wp-content/plugins/ (filesystem, may include inactive)
wp plugin list --format=table
```

→ `db-audit/00-overview.md`

Output: total DB size, top-25 tables ranked by size, list of active plugins, list of all installed plugins (active + inactive). **Flag any single table > 100 MB** — these are usually the cleanup targets.

## BATCH B — Post Bloat (wp_posts + wp_postmeta)

```sql
-- B1. Posts by status & type
SELECT post_type, post_status, COUNT(*) AS cnt
FROM wp_posts
GROUP BY post_type, post_status
ORDER BY cnt DESC;

-- B2. Revisions: count + size impact
SELECT 
  COUNT(*) AS revision_count,
  ROUND(SUM(LENGTH(post_content) + LENGTH(post_content_filtered) + LENGTH(post_excerpt)) / 1024 / 1024, 2) AS content_mb
FROM wp_posts 
WHERE post_type = 'revision';

-- B3. Revisions older than 60 days (safer cleanup target)
SELECT COUNT(*) FROM wp_posts 
WHERE post_type = 'revision' AND post_modified < DATE_SUB(NOW(), INTERVAL 60 DAY);

-- B4. Auto-drafts (almost always safe to delete)
SELECT COUNT(*) FROM wp_posts WHERE post_status = 'auto-draft';

-- B5. Trashed posts (older than 30 days)
SELECT post_type, COUNT(*) 
FROM wp_posts 
WHERE post_status = 'trash' AND post_modified < DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY post_type;

-- B6. Orphaned postmeta (post_id no longer exists)
SELECT COUNT(*) AS orphaned_meta,
       ROUND(SUM(LENGTH(meta_value)) / 1024 / 1024, 2) AS size_mb
FROM wp_postmeta pm
LEFT JOIN wp_posts p ON pm.post_id = p.ID
WHERE p.ID IS NULL;

-- B7. Orphaned attachments — DB records pointing to files that may not exist
-- (list only; do not delete; some "orphans" are template assets registered programmatically)
SELECT ID, post_title, guid 
FROM wp_posts 
WHERE post_type = 'attachment' 
  AND ID NOT IN (SELECT DISTINCT meta_value FROM wp_postmeta WHERE meta_key = '_thumbnail_id' AND meta_value REGEXP '^[0-9]+$')
  AND post_parent = 0
LIMIT 100;
```

→ `db-audit/01-posts.md`

For each finding: count, size impact in MB, sample of 10 rows (with PII redacted), risk classification.

**Stop. Summarize. Wait for "continue."**

## BATCH D — Options Bloat (wp_options) — CRITICAL FOR AUTOLOAD

```sql
-- D1. Total wp_options size + autoload size
SELECT 
  COUNT(*) AS total_rows,
  ROUND(SUM(LENGTH(option_value)) / 1024 / 1024, 2) AS total_mb,
  SUM(autoload = 'yes') AS autoload_rows,
  ROUND(SUM(CASE WHEN autoload = 'yes' THEN LENGTH(option_value) ELSE 0 END) / 1024 / 1024, 2) AS autoload_mb
FROM wp_options;

-- D2. Top 30 largest autoloaded options (these load on EVERY page request)
SELECT 
  option_name,
  LENGTH(option_value) AS size_bytes,
  ROUND(LENGTH(option_value) / 1024, 2) AS size_kb
FROM wp_options 
WHERE autoload = 'yes'
ORDER BY size_bytes DESC
LIMIT 30;

-- D3. Total transients
SELECT 
  COUNT(*) AS transient_rows,
  ROUND(SUM(LENGTH(option_value)) / 1024 / 1024, 2) AS transient_mb
FROM wp_options 
WHERE option_name LIKE '\_transient\_%' OR option_name LIKE '\_site\_transient\_%';

-- D4. Expired transients (safe to delete)
SELECT COUNT(*) AS expired_transients
FROM wp_options a 
INNER JOIN wp_options b ON b.option_name = REPLACE(a.option_name, '_timeout_', '_')
WHERE a.option_name LIKE '\_transient\_timeout\_%'
  AND a.option_value < UNIX_TIMESTAMP();

-- D5. Backup-style options (often left by migration / search-replace plugins)
SELECT option_name, LENGTH(option_value) AS size_bytes
FROM wp_options 
WHERE option_name REGEXP '_backup|_old|_temp|_log_|search_replace_'
ORDER BY size_bytes DESC
LIMIT 20;
```

→ `db-audit/03-options.md`

**This batch is high-impact.** A bloated `wp_options` autoload set is the single biggest cause of slow WordPress sites. Anything > 100 KB autoloaded should be flagged.

For each suspicious option in D2: identify which plugin sets it, whether the plugin is in `active_plugins` (from Batch A), whether it actually needs autoload.

**Stop. Summarize. Wait for "continue."**

## BATCH E — Orphaned Plugin Tables (THE BIG ONE)

This is the most valuable batch for "remove unwanted unlinked plugin data."

```sql
-- E1. All non-core tables
SELECT table_name FROM information_schema.tables 
WHERE table_schema = DATABASE()
  AND table_name NOT IN (
    'wp_commentmeta','wp_comments','wp_links','wp_options',
    'wp_postmeta','wp_posts','wp_termmeta','wp_terms',
    'wp_term_relationships','wp_term_taxonomy','wp_usermeta','wp_users'
  )
ORDER BY table_name;

-- E2. For each non-core table, get size + row count
SELECT 
  table_name,
  table_rows,
  ROUND((data_length+index_length)/1024/1024, 2) AS total_mb
FROM information_schema.tables 
WHERE table_schema = DATABASE()
  AND table_name NOT IN (
    'wp_commentmeta','wp_comments','wp_links','wp_options',
    'wp_postmeta','wp_posts','wp_termmeta','wp_terms',
    'wp_term_relationships','wp_term_taxonomy','wp_usermeta','wp_users'
  )
ORDER BY (data_length+index_length) DESC;
```

For EACH non-core table, classify as:

| Classification | Meaning | Action |
|----------------|---------|--------|
| **ACTIVE** | Belongs to a plugin in `active_plugins` AND in the SOW §7.5 retain list | Keep |
| **ACTIVE — non-§7.5** | Belongs to an active plugin not in SOW §7.5 retain list | Keep, flag for Phase 1 plugin audit |
| **INACTIVE plugin** | Belongs to a plugin installed but not active | Verify with user, candidate for removal |
| **ORPHANED** | Belongs to a plugin no longer in `wp-content/plugins/` | High-priority cleanup candidate |
| **UNKNOWN** | Cannot identify which plugin owns it | Manual review required |

**To classify**, match table-name prefixes against:
- Active plugin slugs (from Batch A)
- Folder names in `wp-content/plugins/`
- Known plugin prefixes (e.g., `wf_` = Wordfence, `gf_` = Gravity Forms, `icl_` = WPML, `actionscheduler_` = Action Scheduler, `wc_` = WooCommerce, `yoast_` = Yoast SEO, `rank_math_` = Rank Math, `wpforms_` = WPForms, `bwf_` = WooFunnels, `mailpoet_` = MailPoet, etc.)

Common orphan patterns at Zycus likely:
- Old import/migration plugin tables (e.g., `wp_aiosp_*`, `wp_duplicator_*`, `wp_updraft_*`)
- Defunct form plugin tables if they switched from Contact Form 7 / WPForms / Ninja Forms to Gravity Forms
- Old SEO plugin tables if they switched between Yoast and Rank Math
- Defunct page builder tables if they migrated from Visual Composer / Divi / Beaver Builder to Elementor
- Old backup plugin tables (UpdraftPlus, BackWPup, BlogVault)
- Old caching plugin tables (W3 Total Cache, WP Super Cache)
- Old chatbot / popup plugin tables (Tidio, OptinMonster)
- Old analytics / heatmap plugin tables (Hotjar, MonsterInsights legacy)
- Old security plugin tables (Sucuri, iThemes Security) — except current Wordfence

→ `db-audit/04-orphaned-tables.md`

**Output:** a table classifying every non-core table, with size, row count, and recommendation. Group by classification.

## BATCH I — Cleanup Plan (Recommended Actions, NOT executed)

Produce `db-audit/08-cleanup-plan.md` with a phased cleanup plan. Every command must be marked as PROPOSED and include `--dry-run` first.

Structure the plan in **risk tiers**:

### Tier 1 — Zero Risk (Safe in any context, but still back up first)

Examples Claude Code should generate based on findings:

```bash
# PROPOSED — DO NOT EXECUTE FROM THIS PROMPT
# Tier 1: zero-risk cleanup

# Expired transients (safe — they were going to expire anyway)
wp transient delete --expired

# Auto-drafts (these are abandoned posts WordPress creates and never cleaned)
wp post list --post_type=any --post_status=auto-draft --format=ids | xargs -r wp post delete --force

# Post revisions older than 90 days (Zycus has versioning in Git; revisions in DB are duplicate)
wp db query "SELECT ID FROM wp_posts WHERE post_type='revision' AND post_modified < DATE_SUB(NOW(), INTERVAL 90 DAY)" --skip-column-names | xargs -r wp post delete --force
```

### Tier 2 — Low Risk (Back up first, verify on test copy)

```bash
# PROPOSED — DO NOT EXECUTE FROM THIS PROMPT
# Tier 2: low-risk cleanup, backup mandatory

# Spam comments
wp comment delete $(wp comment list --status=spam --format=ids)

# Trashed comments
wp comment delete $(wp comment list --status=trash --format=ids)

# Trashed posts older than 30 days
wp db query "SELECT ID FROM wp_posts WHERE post_status='trash' AND post_modified < DATE_SUB(NOW(), INTERVAL 30 DAY)" --skip-column-names | xargs -r wp post delete --force

# Orphaned postmeta
wp db query "DELETE pm FROM wp_postmeta pm LEFT JOIN wp_posts p ON pm.post_id = p.ID WHERE p.ID IS NULL"

# Orphaned usermeta, commentmeta, termmeta — similar pattern
```

### Tier 3 — Review Required (Engineer judgment)

```bash
# PROPOSED — DO NOT EXECUTE — REVIEW EACH TABLE INDIVIDUALLY
# Tier 3: drop orphaned plugin tables identified in Batch E

# Example only — do NOT run blindly. Verify each table belongs to a removed plugin first.
# DROP TABLE wp_old_plugin_table_name;
```

For each Tier 3 candidate, output:
- Table name
- Size
- Likely source plugin
- Evidence it's truly orphaned (plugin folder absent, no `active_plugins` entry, no references in `wp_options`)
- Recommended action (drop, archive, retain, manual review)

### Tier 4 — Autoload Optimization

For each autoloaded option > 100 KB identified in Batch D2:
- Which plugin owns it
- Whether it actually needs autoload (most don't)
- Proposed SQL to change autoload to 'no' (still preserves data):

```bash
# PROPOSED — DO NOT EXECUTE
wp option set <name> "$(wp option get <name>)" --autoload=no
```

### Tier 5 — Discuss with Team

Anything ambiguous goes here — orphan-looking tables that might still be needed, integration data that isn't documented, etc.
